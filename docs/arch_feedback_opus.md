## 3. Decisions accepted from GPT-5.5

### 3.1 UnitOfWork transaction model (GPT 3.1, GPT 11.1, Sonnet Obj. 2)

**Adopt explicit `UnitOfWork`**. Application use cases receive a transaction
and pass it to repositories *and* the AuditSink. Two-phase commit across
Postgres + Qdrant + Redis is not pursued (it doesn't work over those three
systems anyway).

```cpp
struct UnitOfWork {
    DocumentVersionWriter&  document_versions();
    AccessWriter&           access();
    AuditWriter&            audit();
    OutboxWriter&           outbox();
    drogon::Task<Result<void>> commit();
    drogon::Task<void>      rollback();
};
```

All mutations follow the pattern:
1. `auto uow = co_await unit_of_work_factory.begin(ctx);`
2. Repositories write through `uow`.
3. AuditSink writes through `uow`.
4. Outbox event for side effects (Qdrant resync, Redis invalidation) written
   through `uow`.
5. `co_await uow.commit();` — atomic in Postgres; everything else is
   eventually consistent via the outbox worker.

### 3.2 Outbox pattern (GPT 2.2, 6.1, 11.2)

**Adopt the transactional outbox** for every state change that requires a
Qdrant or Redis follow-up. New schema migration (V015) needed:

```sql
CREATE TABLE outbox_events (
    id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    company_id         UUID NOT NULL REFERENCES companies(id),
    aggregate_id       UUID,
    job_type           TEXT NOT NULL,
    job_schema_version SMALLINT NOT NULL,
    payload            JSONB NOT NULL,
    idempotency_key    TEXT NOT NULL,
    claimed_at         TIMESTAMPTZ,
    claimed_by         TEXT,
    completed_at       TIMESTAMPTZ,
    attempt_count      INT NOT NULL DEFAULT 0,
    last_error         TEXT,
    created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (company_id, job_type, idempotency_key)
);
```

Workers use `SELECT ... FOR UPDATE SKIP LOCKED` to claim. Polling fallback
in scheduler re-enqueues jobs stuck >N minutes. The `job_schema_version`
field handles rolling deploys (GPT 2.1).

### 3.3 Executor split (GPT 4.4, 11.3)

**Accept "one executor pool" was wrong.** Drogon's threading model has:

- **I/O event loop pool** (sized to cores) — HTTP, Postgres, Redis, Qdrant calls
- **Worker thread pool** (Drogon's `getIOThreadPool` / `getRootThreadPool`) — CPU-bound work
- **Blocking pool** — wrap any blocking native call (PDF parse, JSON of large
  documents, JWT crypto where the lib is sync)
- **GPU/embedding semaphore** — limit concurrency for llama-server hits to
  match its actual parallel decoding capacity

Named helpers (`run_on_cpu_pool(...)`, `run_blocking(...)`) make every pool
crossing explicit. Add Prometheus queue-depth metrics per pool.

### 3.4 EvidenceGate type-system enforcement (GPT 5.1, 5.5, 11.5)

**Strongly accept.** This is the kind of contract C++ does better than most
languages: make the policy violation a *compile error*, not a code-review issue.

```cpp
class ChunkCandidate { /* raw from any source */ };

// Construction is private; only EvidenceGate can produce these.
class AllowedCandidate {
    friend class EvidenceGate;
    AllowedCandidate(ChunkCandidate, AccessEnvelope);
public:
    // accessors only
};

class EvidenceGate {
public:
    drogon::Task<std::vector<AllowedCandidate>>
    filter(const RequestContext&, std::vector<ChunkCandidate>) const;
};

// Reranker takes AllowedCandidate, not ChunkCandidate:
class Reranker {
public:
    virtual drogon::Task<std::vector<AllowedCandidate>>
    rerank(const RequestContext&, std::vector<AllowedCandidate>) = 0;
};
```

A reranker that tries to widen the candidate set fails to compile. A stage
that fetches sibling chunks via section_id must run them through
EvidenceGate before they can enter the reranker. Property tests assert
"no `ChunkCandidate` ever reaches the context builder without traveling
through EvidenceGate".

### 3.5 Pipeline ordering contract (GPT 5.3)

**Accept.** Pipeline builder validates the stage graph against a fixed
schema:

```
Mandatory precondition:    BuildAccessPolicy
Candidate-producing slots: VectorRetrieval | BM25 | CuratedAnswer (any order)
Mandatory after each:      EvidenceGate
Optional enrichment:       SectionExpansion | Reranker (in this slot)
Mandatory finalization:    ContextBuilder + CitationBuilder + AuditEnvelope
```

Tenant config can swap implementations within slots but cannot remove
mandatory stages or move enrichment before security gates. The pipeline
builder rejects invalid configurations at startup; misconfiguration is a
boot-time failure, not a runtime security bug.

### 3.6 "Qdrant is index, Postgres is evidence" (GPT 6.1)

**Strongly accept.** Add as an explicit invariant:

- Qdrant returns point IDs and similarity scores.
- Postgres hydrates exact chunk text, document version, section, sensitivity
  label, lifecycle timestamps from `document_chunks` joined on
  `document_versions`.
- ContextBuilder reads chunk text from Postgres hydration, never from Qdrant
  payload.
- A Qdrant payload field that disagrees with Postgres is a *signal to
  resync*, not authoritative data. Resync job is enqueued in the outbox.

### 3.7 Access epoch for cache invalidation (GPT 6.2)

**Accept with caveat.** The reverse-index Sets pattern (V008) is good for
the common case of granular invalidation. The `access_epoch` pattern is
better for high-risk revocations where stale data is unacceptable. Use
both:

- Reverse-index Sets for normal grant/revoke (already in V008).
- `lr:access_epoch:{company_id}` integer, bumped on
  emergency-revoke / break-glass actions. `lr:eff` values include the epoch
  they were computed under; mismatch → cache miss.

The epoch path is opt-in per action, not the default.

### 3.8 Constraint-name to domain-error mapping (GPT 3.3)

**Accept.** Add a `pg_error_mapper` adapter that converts Postgres SQLSTATE
+ constraint names to typed application errors:

| Constraint name | App error |
|---|---|
| `document_versions_one_active_per_doc_uidx` | `Error::Conflict("two active versions")` |
| `memberships_user_org_uidx` | `Error::Conflict("membership exists")` |
| `chat_turns_rag_sources_array_chk` | `Error::InvalidState("rag_sources malformed")` |
| `org_unit_self_loop_chk` | `Error::InvalidState("org_unit self-loop")` |

The mapper is exhaustive — unmapped constraint violations log a warning
and surface as `Error::DatabaseError`, with a CI test that fails if a new
constraint is added without a mapping.

### 3.9 Narrow ports over god interfaces (GPT 3.2)

**Accept.** Split monolithic ports per-aggregate:

- `DocumentVersionReader`, `DocumentVersionWriter`
- `WikiPageReader`, `WikiPagePublisher`
- `AccessScopeReader`
- `AuditWriter`
- `OutboxWriter`, `OutboxClaimer`

If a port grows past ~6 methods, split it.

### 3.10 Cancellation safety with RAII (GPT 4.5, 4.6)

**Accept.** Three rules:

1. Every resource acquisition (Postgres transaction, Redis semaphore,
   job lease) is wrapped in an RAII guard with a destructor that cleans
   up. Drogon coroutines respect destructor invocation on
   cancellation.
2. After a state mutation has begun in a transaction, commit + audit +
   outbox are *shielded from cancellation*. Either all three succeed or
   none do.
3. Cancellation signals distinguish *user cancellation* (client
   disconnect) from *server shutdown* (SIGTERM drain). The latter is
   given a longer grace window.

### 3.11 Observability redaction (GPT 5.6)

**Accept.** Logs and traces include: IDs, counts, hashes, token lengths,
model names, latencies. They do NOT include: raw prompt text, completion
text, chunk text, user questions, document content. Debug capture
requires a per-tenant flag in `companies.settings` and writes to a
separate quarantined log sink, not the main observability pipeline.

### 3.12 Other accepted items

- GPT 2.3: ingest idempotency keys spec'd (chunk_index unique;
  Qdrant point ID = `uuid_v5(chunk_id || embedding_model_id)`).
- GPT 2.4: scheduler advisory locks via `pg_try_advisory_lock(hashtext(job_name))`.
- GPT 2.5: per-tenant ingest fairness — separate Redis lists per tenant,
  worker fetches round-robin. Hard ceiling on concurrent ingest per tenant
  configured in `companies.settings`.
- GPT 5.2: terminal curated answers still pass through `AnswerFinalizerStage`

- GPT 7.1: framework-independent DTOs at application layer; no Drogon types

- GPT 8.1: `TenantResolver` port spec'd.
- GPT 8.2: workers use same repository library, not direct SQL; service
  identity in audit log as `actor_type='service'`, `actor_name='wikore-ingest'`.
- GPT 9.1: tenant labels only on business metrics; latency metrics labeled

- GPT 9.2: degraded-but-readyz separation.
- GPT 9.3: shutdown releases leases + semaphores.

## 4. Where I push back (small list)

### 4.1 GPT 3.4 — "swappable adapters is oversold"

**Partially accept.** The framing in the original architecture document
was too optimistic — Qdrant payload-filter semantics are not portable to
Weaviate or Vespa without translation logic. But the hexagonal layering
*is* still useful even when adapters are not 1:1 swappable: it makes the
contract surface explicit, makes mocking cheap, and lets you stage a
migration over months rather than weeks.

Revised wording: "Adapters isolate vendor APIs. Replacing one vendor with
another is *new adapter + new mapping logic + integration test rewrite*,
not a domain rewrite. This is days-to-weeks, not days." Drop the "one
new adapter" claim from §3 of architecture.md.

### 4.2 GPT 4.3 — libpqxx async spike

**Moot, accept the underlying concern.** Since we're dropping libpqxx for
Drogon's built-in async Postgres (decision L1), the specific concern about
libpqxx async maturity goes away. The underlying concern — that DB calls
truly be async — still applies and Drogon's PG client documentation should
be the verification target. A "100 concurrent simple queries, no executor
starvation" spike is still in iteration 0.

### 4.3 Sonnet's "two SQL libraries" → settled differently

Sonnet recommended dropping sqlpp23 and using libpqxx + Drogon ORM.
Decision L1 means we drop libpqxx too. End state: Drogon ORM for app
code, raw SQL strings through Drogon's async PG client for the hot
retrieval path. One library, two idioms — but the same vendor.

### 4.4 GPT 13 — "do not approve until §12 contracts added"

**Accept the spirit.** This document is the §12-contract addendum. With
the deltas in this file applied, the architecture is ready as a coding
contract. No coding before iteration 0 of the development plan starts
(which is small and self-contained anyway).

## 5. Newly-required decisions (raised by both reviewers, not yet decided)

### 5.1 Embedding model + Qdrant collection name

**Proposal:**

| Concern | Choice |
|---|---|
| Embedding model | **bge-m3** (1024 dims, multilingual, supports up to 8192 tokens) served via llama-server |
| Reranker model | **bge-reranker-v2-m3** served via llama-server's `/v1/rerank` |
| Qdrant collection name | `wikore_chunks_v1` (suffix bumped on model change) |
| Vector dimension | 1024 |
| Distance metric | Cosine |
| Payload schema version | starts at 1; bumped per V003 contract |

bge-m3 is the strongest open-weight multilingual embedder at this size as
of mid-2026, supports 8k context (matches typical chunk + section
expansion), and pairs naturally with bge-reranker-v2-m3 for two-stage
retrieval. text-embedding-3-large from OpenAI is an alternative if
customers want closed-source; the schema dimension constraint means we'd
need a separate collection per provider.

**Action required:** confirm or override. Locked-in before iteration 1.

### 5.2 Chunk budget per document

**Proposal:** target ~600 character chunks with 100-char overlap, max
~2000 chars per chunk for hierarchical structure-preserving (K5) chunks.
This matches typical RAG chunk sizes and fits comfortably in bge-m3's
context.
